# CS300-DSA-Analysis-and-Design
Coursework from CS300 Southern New Hampshire University

Project Reflection

These projects were all about figuring out how different data structures actually work in real programs, not just in theory. In Project One, I had to compare run‑time and memory use, which at first felt a little abstract, but it really helped later, escpecially aftr all of the amazing and well thought out critiques I recieved from Dr. Webb. Project Two was where everything came together, I built a course planner that loads a CSV file, stores the courses, sorts them, and prints out the results. I ended up using a hash table (unordered_map) because of what I used and learned earlier, and it turned out to be a pretty good choice for fast lookups and since it was something I undesrtood.
My approach was honestly a bit of try something, see why it breaks, fix it, repeat. I tried to rely on what I learned about Big‑O to decide how to structure the program, and it finally clicked why picking the right data structure matters. Before this class I probably would’ve just thrown everything into a vector and sorted it every time, but using the hash table made the rest of the program easier I found out, though there are different ways I could do it.
I definitely hit some roadblocks. My CSV parsing was rough at first, I had to write my own trim() and even a quick CSV splitter because weird spaces kept breaking things. I also ran into issues where my course lookups were failing just because the case didn’t match, so I wrote an up() function to force everything to uppercase. And the sorting threw me at the beginning because I forgot you can’t sort an unordered_map directly. I ended up building a separate vector of course pointers and sorting that instead. None of this was smooth, but figuring out each problem helped me understand the logic way better.
Working through the project changed how I think about designing programs. I’m starting to see how much planning ahead matters. Instead of just coding until something works (which I’ve definitely done plenty of times), I’m noticing that thinking through the structure first actually saves time. Even things like how I printed prerequisites made me think more about data flow and edge cases, like what to do if a prereq isn’t even in the file.
I also feel like my code got a little more readable and organized as I worked through this. I separated things into helper functions, tried to name stuff better, and even added debugging checks like lastLoadCount to help myself out. I still rewrite things more than I’d like to, but I can tell I’m getting better at making my programs more maintainable and less confusing when I come back to them.
Overall, these projects weren’t the easiest, but they pushed me to understand data structures in a more practical way, and I’m walking away with a better sense of how to build programs that are cleaner and actually make sense.
